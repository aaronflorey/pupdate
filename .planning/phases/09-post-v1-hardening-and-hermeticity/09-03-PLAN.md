# Plan 09-03 - Add timeout/injection for git submodule freshness

## Objective

Prevent git submodule freshness checks from hanging hook-driven runs by making the `git submodule status --recursive` path injectable and timeout-bounded.

## Scope

- Add a test seam for the submodule status command in `internal/freshness`.
- Execute the git status check with an explicit timeout using stdlib context/process APIs.
- Preserve the existing non-blocking skip/error reporting contract when the check fails or times out.

## Tasks

1. Refactor the submodule status call site to depend on an injectable runner rather than direct `exec.Command` construction.
2. Add timeout handling that returns a surfaced freshness reason instead of hanging the run path.
3. Extend freshness and command tests to cover timeout/failure behavior and successful drift detection.

## Verification

- `go test ./internal/freshness ./cmd/pupdate -count=1`

## Done When

- Git freshness checks can be unit tested without invoking real git.
- Slow or stuck submodule status calls fail fast with visible non-blocking status output.
