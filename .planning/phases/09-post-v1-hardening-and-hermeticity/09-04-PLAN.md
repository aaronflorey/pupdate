# Plan 09-04 - Reduce hot-path lockfile hashing cost

## Objective

Cut unnecessary work in freshness evaluation by avoiding full lockfile content hashing when lockfile metadata proves files are unchanged.

## Scope

- Focus on the lockfile-hash hot path in `internal/freshness`.
- Keep state semantics correct for changed, missing, and newly added lockfiles.
- Avoid broad schema redesign unless correctness requires a small state extension.

## Tasks

1. Measure the current hashing flow and identify the minimal metadata needed for a safe unchanged-file fast path.
2. Implement the reduced-cost comparison path while preserving exact change detection when metadata differs.
3. Add regression tests for unchanged, changed, renamed, and missing lockfile cases.

## Verification

- `go test ./internal/freshness -count=1`
- `go test ./cmd/pupdate -count=1`

## Done When

- Unchanged lockfiles avoid unnecessary full hashing on the common path.
- Freshness decisions remain correct across lockfile change scenarios.
