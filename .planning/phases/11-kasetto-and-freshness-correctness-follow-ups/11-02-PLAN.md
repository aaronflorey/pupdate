# Plan 11-02 - Align Kasetto detection and execution around explicit local configs

## Objective

Ensure Kasetto execution uses detected local `kasetto.yaml` or `kasetto.yml` files explicitly and that lock-only detections do not fall back to global/default config behavior.

## Scope

- Audit how `internal/detection` reports Kasetto lock/config signals and how `cmd/pupdate` consumes those signals during execution.
- Pass detected local Kasetto config files explicitly to execution when they are present.
- Change the lock-only Kasetto path so it does not run through global/default Kasetto config discovery.
- Add targeted regression coverage in detection and command tests for `kasetto.yaml`, `kasetto.yml`, and lock-only scenarios.

## Tasks

1. Confirm the current Kasetto detection contract for config-plus-lock and lock-only projects.
2. Implement the minimal contract changes needed so execution can distinguish explicit local config files from lock-only detections.
3. Re-run focused `cmd/pupdate` and detection tests for both local config filename variants and the lock-only case.

## Verification

- `go test ./cmd/pupdate ./internal/detection -count=1`

## Done When

- `kasetto.yaml` and `kasetto.yml` detections are passed explicitly into Kasetto execution.
- Lock-only Kasetto detection no longer falls back to global/default config behavior.
