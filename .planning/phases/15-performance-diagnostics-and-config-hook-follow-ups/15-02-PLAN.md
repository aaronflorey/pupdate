# Plan 15-02 - Add a diagnostic command

## Objective

Introduce a dedicated `pupdate status` or `pupdate doctor` command that helps users understand detection, freshness, configuration, environment, and state issues without reading internal files directly.

## Scope

- Cover the diagnostic surface needed to explain why `pupdate` would run, skip, or fail in the current directory.
- Decide during execution whether `status` or `doctor` best matches the intended UX and naming conventions.
- Keep the first version read-only and troubleshooting-oriented.

## Tasks

1. Audit the current troubleshooting gaps in `run`, state, config, and PATH-resolved manager detection.
2. Define the minimum diagnostic output contract for the chosen command name and scope.
3. Add command coverage and docs that keep diagnostics clear without duplicating all of `run` behavior.

## Verification

- Command-level tests for representative success, skip, misconfiguration, and missing-manager scenarios.
- Manual review that the output explains the current directory state without mutating it.

## Done When

- Users can run one diagnostic command to understand the current project's detection and freshness state.
- The command improves troubleshooting without changing normal auto-update behavior.
