# Plan 15-06 - Add an opt-in async/background hook mode

## Objective

Provide an optional shell-hook mode that runs updates asynchronously in the background so users can opt into lower perceived navigation latency without changing the default foreground model.

## Scope

- Keep the mode explicitly opt-in and preserve current foreground behavior as the default.
- Define how status, errors, and overlapping runs should behave when work is detached from the interactive shell transition.
- Avoid broad shell-integration rewrites beyond what the async mode requires.

## Tasks

1. Define the user-facing contract for enabling and disabling background hook execution.
2. Plan the minimal shell and CLI coordination needed for background runs, visibility, and conflict avoidance.
3. Add tests and docs that prove the async mode is optional, understandable, and non-destructive to existing hook behavior.

## Verification

- Shell-integration and command-level coverage for enabled and disabled async mode behavior.
- Manual review of status/error visibility expectations for detached runs.

## Done When

- Users have a clearly documented opt-in path for background hook execution.
- Default hook behavior remains unchanged while async mode semantics are explicit and testable.
