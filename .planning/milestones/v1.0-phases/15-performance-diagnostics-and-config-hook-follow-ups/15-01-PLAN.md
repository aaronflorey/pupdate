# Plan 15-01 - Reuse stored lockfile metadata safely

## Objective

Reduce hot-path freshness cost by reusing stored lockfile metadata to skip full hashing only when the unchanged-file conclusion is provably safe.

## Scope

- Focus on the freshness hot path and the persisted state fields needed to support a safe shortcut.
- Preserve the correctness guarantees restored by Phase 11 plan 03.
- Avoid broader state-schema redesign unless a small extension is required.

## Tasks

1. Confirm which stored lockfile metadata is already available and what additional metadata, if any, is required for a safe fast path.
2. Define the precise conditions under which metadata can short-circuit rehashing without masking real content changes.
3. Add targeted tests and measurements that prove the shortcut is both correct and materially cheaper on the common path.

## Verification

- Package-scoped freshness tests covering unchanged, changed, renamed, and removed lockfile cases.
- Focused benchmark or timing evidence showing reduced hot-path work when the safe shortcut applies.

## Done When

- Common unchanged-file freshness checks can skip unnecessary hashing only under explicitly safe conditions.
- The resulting behavior does not reintroduce the metadata-only correctness bug closed in Phase 11.
