# Plan 11-03 - Remove or replace metadata-only lockfile hash reuse

## Objective

Restore content-correct freshness skip decisions by removing or replacing the metadata-only hash reuse fast path in `internal/freshness`.

## Scope

- Audit the unchanged-file fast path in `internal/freshness/engine.go` and the tests added around metadata reuse.
- Remove the optimization if no correctness-preserving replacement is available with small scope.
- Add or update regression coverage for same-metadata/different-content behavior so skip decisions remain content-correct.

## Tasks

1. Confirm the exact metadata-only reuse contract in `hashMatchedFiles` and identify the narrowest safe replacement option.
2. Implement either a correctness-preserving replacement or a straightforward removal of the unsafe reuse path.
3. Re-run focused freshness tests covering unchanged files, changed files, renamed files, and metadata-collision scenarios.

## Verification

- `go test ./internal/freshness -count=1`

## Done When

- Freshness no longer treats matching metadata alone as sufficient proof that lockfile contents are unchanged.
- Regression tests cover a case where metadata matches but content differs.
