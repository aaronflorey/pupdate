# P2-T1: Extract shared run/status preflight collection

Status: complete
Type: implementation
Phase: 2 shared preflight flow
Depends on: P1-R1
PRD refs: `.planning/update.md` -> `Maintainability` / `6. Centralize shared run and status preflight flow`
Approved-plan refs: `.planning/quick/phase-15-performance-diagnostics-config-hook-planning.md`

## Objective

Extract a shared preflight collection layer used by both `run` and `status` without changing their distinct behaviors.

## Context

- The PRD identifies duplicated setup and decision logic across `run` and `status`.
- The shared flow includes config loading, root checks, `.pupignore`, detection, state loading, and freshness evaluation.
- Later status-guidance work should build on a single normalized preflight result rather than duplicating more branching.

## Expected files

Create or modify only these files unless a stop condition applies:

- `cmd/pupdate/run_execution.go`: replace duplicated preflight setup with the shared helper.
- `cmd/pupdate/status.go`: replace duplicated preflight setup with the shared helper.
- `cmd/pupdate/run_test.go`: update or add regression coverage for `run` behavior.
- `cmd/pupdate/status_test.go`: update or add regression coverage for `status` behavior.
- `cmd/pupdate/preflight.go`: add the shared collection/preflight layer.

## Required changes

1. Define a shared helper or data structure that collects the common preflight inputs and decisions.
2. Refactor `run` and `status` to use that shared path while keeping their distinct output and side-effect behavior.
3. Add regression coverage proving both commands still respect home-directory, configured-root, `.pupignore`, detection, and state/freshness behavior.

## Non-goals

- Do not add new user-facing remediation text in this task.
- Do not change detection depth, config schema, or manager resolution behavior.

## Acceptance criteria

- `run` and `status` both use a common preflight collection layer.
- Existing skip/readiness semantics stay unchanged apart from internal refactoring.
- Regression tests cover the shared preflight path for both commands.

## Checks

Run or inspect:

- `go test ./cmd/pupdate -count=1`
- Manual inspection that the shared helper owns the duplicated setup logic

## Review instructions

The reviewer must verify:

- The implementation matches this brief and does not add unrequested behavior.
- All acceptance criteria are satisfied.
- Required tests or checks pass, or failures are explained and unrelated.
- No later-phase work was pulled into this task.

## Stop conditions

Stop and ask the orchestrator or user if:

- The refactor would require moving logic into packages outside `cmd/pupdate`.
- Existing tests reveal behavior differences that are product decisions rather than refactor defects.
- The work would require modifying unrelated command surfaces not listed above.
