# P2-T2: Add action-oriented status remediation guidance

Status: complete
Type: implementation
Phase: 2 shared preflight flow
Depends on: P2-T1
PRD refs: `.planning/update.md` -> `UX and Adoption` / `8. Make status more action-oriented`
Approved-plan refs: `.planning/quick/phase-15-performance-diagnostics-config-hook-planning.md`

## Objective

Add action-oriented remediation guidance to `pupdate status` for the blocked states already detected by the shared preflight flow.

## Context

- The PRD calls out missing remediation text for states such as missing managers on `PATH`, foreground hook mode, and configured-root exclusions.
- Phase 2 already normalizes preflight decisions, which should be reused instead of re-deriving the same reasons in display code.
- The feature is diagnostic UX only; it must not change `run` behavior.

## Expected files

Create or modify only these files unless a stop condition applies:

- `cmd/pupdate/status.go`: add targeted guidance text for current blocked or surprising states.
- `cmd/pupdate/status_test.go`: add or update tests for the new guidance output.

## Required changes

1. Identify the current blocked or confusing `status` states that should emit remediation guidance.
2. Add concise, deterministic guidance text for those states without changing the existing status facts.
3. Add tests that pin the new guidance output for representative cases.

## Non-goals

- Do not add a new `doctor` command.
- Do not modify the underlying run decision logic in this task.

## Acceptance criteria

- `pupdate status` includes concrete next-step guidance for the supported blocked states covered by the PRD.
- The guidance text is deterministic and test-covered.
- Existing status fields remain readable and do not change meaning.

## Checks

Run or inspect:

- `go test ./cmd/pupdate -count=1`
- Manual inspection of `status` output fixtures or assertions in `cmd/pupdate/status_test.go`

## Review instructions

The reviewer must verify:

- The implementation matches this brief and does not add unrequested behavior.
- All acceptance criteria are satisfied.
- Required tests or checks pass, or failures are explained and unrelated.
- No later-phase work was pulled into this task.

## Stop conditions

Stop and ask the orchestrator or user if:

- A proposed guidance string would commit the product to a new support policy or config model.
- The desired guidance cannot be expressed from the data already produced by the shared preflight layer.
- The work would require changing files outside the allowed list.
