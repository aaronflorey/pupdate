# P3-T1: Define validated workspace_globs config support

Status: pending
Type: implementation
Phase: 3 monorepo scan expansion
Depends on: P1-R1
PRD refs: `.planning/update.md` -> `Product Gaps` / `2. Revisit detection depth for common monorepo layouts`
Approved-plan refs: `.planning/quick/phase-15-performance-diagnostics-config-hook-planning.md`

## Objective

Define validated `workspace_globs` config support and surface it through `pupdate config`.

## Context

- The PRD says broader monorepo scanning must stay opt-in to preserve low-latency defaults.
- The user chose configurable glob patterns instead of fixed extra roots or a max-depth setting.
- Config parsing and `pupdate config` output must become authoritative before detection starts consuming the new setting.

## Expected files

Create or modify only these files unless a stop condition applies:

- `cmd/pupdate/config.go`: add the `workspace_globs` config field and validation/normalization behavior.
- `cmd/pupdate/config_test.go`: cover parsing, defaults, and invalid values.
- `cmd/pupdate/config_cmd.go`: print the resolved `workspace_globs` setting.
- `cmd/pupdate/config_cmd_test.go`: cover the config command output.

## Required changes

1. Add `workspace_globs` to the user config model with deterministic default behavior when the field is missing.
2. Validate and normalize the configured glob values so later detection logic receives a safe input shape.
3. Expose the configured and resolved `workspace_globs` values through `pupdate config` output and tests.

## Non-goals

- Do not change detection traversal behavior in this task.
- Do not document the new setting in the README yet.

## Acceptance criteria

- Missing config keeps the current shallow scan default with no implicit globs.
- Invalid `workspace_globs` values are handled consistently and test-covered.
- `pupdate config` reports the new setting in a deterministic format.

## Checks

Run or inspect:

- `go test ./cmd/pupdate -count=1`
- Manual inspection of config parsing and config-command tests

## Review instructions

The reviewer must verify:

- The implementation matches this brief and does not add unrequested behavior.
- All acceptance criteria are satisfied.
- Required tests or checks pass, or failures are explained and unrelated.
- No later-phase work was pulled into this task.

## Stop conditions

Stop and ask the orchestrator or user if:

- The chosen glob syntax is not supported cleanly by existing Go stdlib or current repo conventions.
- Validation would require a new dependency not already approved.
- The work would require changing detection files before the config contract is settled.
