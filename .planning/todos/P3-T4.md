# P3-T4: Define folder blacklist config support

Status: pending
Type: implementation
Phase: 3 monorepo scan expansion
Depends on: P1-R1
PRD refs: inserted user idea on 2026-06-04: "allow blacklisting folders so we don't enter them. eg. `blah` so ./blah/ or ./foo/blah would be blacklisted."
Approved-plan refs: `.planning/quick/phase-15-performance-diagnostics-config-hook-planning.md`

## Objective

Define validated folder blacklist config support for exact directory-name matches and surface it through `pupdate config`.

## Context

- The user approved a new detection exclusion feature that should apply across all detection paths, not only opt-in workspace expansion.
- The user clarified that each blacklist entry is an exact directory-name segment match, so `blah` skips `./blah` and `./foo/blah`, but not unrelated names.
- The config contract must be settled before detection logic starts consuming the new setting.

## Expected files

Start with these expected files. You may create or modify another file only when it is directly required for this TODO, consistent with the approved plan, and documented in your report:

- `cmd/pupdate/config.go`: add the folder blacklist config field plus validation and normalization behavior.
- `cmd/pupdate/config_test.go`: cover parsing, defaults, and invalid values for the new field.
- `cmd/pupdate/config_cmd.go`: print the resolved folder blacklist setting.
- `cmd/pupdate/config_cmd_test.go`: cover the config command output for the new field.

## Required changes

1. Add a folder blacklist field to the user config model with deterministic default behavior when the field is missing.
2. Validate and normalize the configured values as exact directory-name entries so later detection logic receives a safe input shape.
3. Expose the configured and resolved folder blacklist values through `pupdate config` output and tests.

## Non-goals

- Do not change detection traversal behavior in this task.
- Do not document the new setting in the README yet.
- Do not broaden the config contract into glob or path-pattern matching.

## Acceptance criteria

- Missing config keeps the current detection behavior with no implicit blacklist entries.
- Invalid blacklist values are handled consistently and test-covered.
- `pupdate config` reports the new setting in a deterministic format.
- The config contract makes clear that entries are exact directory-name matches, not globs.
