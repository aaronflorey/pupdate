# P1-T1: Make async hook mode the default

Status: pending
Type: implementation
Phase: 1 hook and platform contract
Depends on: none
PRD refs: `.planning/update.md` -> `Highest ROI` / `1. Make hook execution non-blocking by default, or align the docs`
Approved-plan refs: `.planning/quick/phase-15-performance-diagnostics-config-hook-planning.md`

## Objective

Change `pupdate init` so async hook mode is the default and the generated hook help/tests match that contract.

## Context

- The PRD identifies a mismatch between the product promise of non-blocking directory entry and the current foreground default.
- The user chose the product direction to make async the default instead of only tightening documentation.
- Existing async hook support and lock behavior already exist, so this task should reuse that flow rather than invent a second background mechanism.

## Expected files

Create or modify only these files unless a stop condition applies:

- `cmd/pupdate/init.go`: change the default mode and the related flag/help text.
- `cmd/pupdate/init_test.go`: update expectations for generated snippets and defaults.
- `cmd/pupdate/hook_test.go`: update any tests that assume foreground is the generated default.

## Required changes

1. Change the `init` command's default `--mode` value from `foreground` to `async`.
2. Update the command help text so it describes async as the default and foreground as the explicit opt-out mode.
3. Adjust command-level tests so they prove the generated shell hook defaults to the async path.

## Non-goals

- Do not change the underlying async lock implementation in `hook.go`.
- Do not update README prose in this task.

## Acceptance criteria

- Running `pupdate init --shell <shell>` without `--mode` now emits an async hook snippet.
- Running `pupdate init --shell <shell> --mode foreground` still emits the synchronous hook variant.
- Tests covering `init` and hook snippet behavior pass with the new default.

## Checks

Run or inspect:

- `go test ./cmd/pupdate -count=1`
- Manual inspection of generated snippet expectations in `cmd/pupdate/init_test.go`

## Review instructions

The reviewer must verify:

- The implementation matches this brief and does not add unrequested behavior.
- All acceptance criteria are satisfied.
- Required tests or checks pass, or failures are explained and unrelated.
- No later-phase work was pulled into this task.

## Stop conditions

Stop and ask the orchestrator or user if:

- Another command or public API depends on foreground remaining the default.
- The generated snippet path would require changing files outside the allowed list.
- Existing tests reveal shell-specific async behavior that changes the approved product decision.
