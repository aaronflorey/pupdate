# P4-T3: Add focused CLI integration coverage

Status: pending
Type: implementation
Phase: 4 docs, tooling, and CLI confidence
Depends on: P2-R1, P1-R1
PRD refs: `.planning/update.md` -> `Developer Experience` / `10. Add a small end-to-end CLI test layer`
Approved-plan refs: none

## Objective

Add a focused CLI integration test layer for `run`, `status`, `init`, and async hook lock lifecycle behavior.

## Context

- The PRD identifies cross-module risk across config, detection, freshness, manager selection, and command output.
- Earlier phases change hook defaults and shared preflight behavior, so the integration suite should exercise those user-visible flows together.
- The suite should stay lightweight and use temporary directories plus stubbed command resolution rather than real package-manager installs.

## Expected files

Create or modify only these files unless a stop condition applies:

- `cmd/pupdate/cli_integration_test.go`: add the focused end-to-end command coverage.
- `cmd/pupdate/test_env_test.go`: extend shared test helpers if the new integration suite needs them.

## Required changes

1. Add integration-style coverage for `pupdate run`, `pupdate status`, and `pupdate init` using isolated temporary directories.
2. Cover the async hook lock lifecycle at the CLI level without depending on real background package-manager side effects.
3. Keep the suite deterministic by stubbing command resolution or child-command execution as needed.

## Non-goals

- Do not add a full external end-to-end harness or shell-based test framework.
- Do not expand the suite into unrelated feature areas beyond the PRD candidates.

## Acceptance criteria

- The repository has a lightweight CLI integration test layer covering the listed high-value flows.
- The tests run under `go test ./... -count=1` without requiring real dependency-manager binaries.
- The async hook lock lifecycle has explicit regression coverage at the CLI level.

## Checks

Run or inspect:

- `go test ./... -count=1`
- Manual inspection that the tests isolate filesystem and command-resolution side effects

## Review instructions

The reviewer must verify:

- The implementation matches this brief and does not add unrequested behavior.
- All acceptance criteria are satisfied.
- Required tests or checks pass, or failures are explained and unrelated.
- No later-phase work was pulled into this task.

## Stop conditions

Stop and ask the orchestrator or user if:

- The current test seams cannot support deterministic CLI integration coverage without a broader testability refactor.
- The work would require real network access or real package-manager binaries.
- The implementation would need to modify files outside the allowed list.
