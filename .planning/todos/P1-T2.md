# P1-T2: Align shipped platform support and unsupported-OS docs

Status: pending
Type: implementation
Phase: 1 hook and platform contract
Depends on: P1-T1
PRD refs: `.planning/update.md` -> `Reliability` / `3. Fix Windows background hook lock handling`, `4. Add Windows CI coverage or stop releasing Windows artifacts`; `Performance` / `5. Improve lockfile hash reuse on non-Linux/macOS platforms`
Approved-plan refs: `.planning/quick/post-v1-hardening-phase-planning.md`

## Objective

Remove Windows from the shipped support surface and document the unsupported-OS freshness boundary.

## Context

- The PRD highlights a Windows stale-lock correctness gap and a release-vs-test mismatch.
- The user chose to drop Windows releases instead of planning Windows bug fixes and CI coverage.
- The user also chose documentation over new platform-specific file identity work for unsupported operating systems.

## Expected files

Create or modify only these files unless a stop condition applies:

- `.goreleaser.yaml`: remove Windows release targets.
- `README.md`: update supported-platform and freshness-performance wording so unsupported OS behavior is explicit.
- `CONTRIBUTING.md`: update contributor expectations if platform support wording needs to match the README.

## Required changes

1. Remove Windows from the release artifact matrix so shipped binaries align with tested platforms.
2. Update user-facing docs to stop implying Windows support.
3. Document that unsupported operating systems may rehash unchanged lockfiles more often because file-identity shortcuts are only implemented on supported platforms.

## Non-goals

- Do not add Windows-specific process-liveness code.
- Do not add new CI jobs in this task.

## Acceptance criteria

- `.goreleaser.yaml` no longer publishes Windows binaries.
- README support messaging no longer promises Windows support.
- README or contributing docs explain the unsupported-OS freshness limitation without changing runtime behavior.

## Checks

Run or inspect:

- Manual inspection of `.goreleaser.yaml` build targets
- Manual inspection of README and contributing support wording

## Review instructions

The reviewer must verify:

- The implementation matches this brief and does not add unrequested behavior.
- All acceptance criteria are satisfied.
- Required tests or checks pass, or failures are explained and unrelated.
- No later-phase work was pulled into this task.

## Stop conditions

Stop and ask the orchestrator or user if:

- Another release or packaging file still advertises Windows support.
- Unsupported-OS documentation would require product-code changes to stay truthful.
- The work would require editing CI or runtime code outside the allowed files.
