Status: pending

## Objective

Apply the folder blacklist across all detection paths, including the default shallow scan and configured workspace expansions.

## Context

- The user approved blacklist scope across all detection, so matching directories must be skipped in both the current shallow walk and any workspace-glob expansion paths.
- Each configured entry is an exact directory-name match, not a glob or full path pattern.
- Detection changes must preserve existing package-root semantics and default latency expectations outside the excluded directories.

## Expected files

Start with these expected files. You may create or modify another file only when it is directly required for this TODO, consistent with the approved plan, and documented in your report:

- `internal/detection/detector.go`: apply folder blacklist checks before entering matched directories across detection paths.
- `internal/detection/detector_test.go`: add cases for blacklisted directories in shallow scan and nested paths.
- `internal/detection/detector_benchmark_test.go`: extend or inspect guardrail coverage if blacklist checks change benchmark setup.
- `cmd/pupdate/run_execution.go`: thread resolved blacklist config into detection.
- `cmd/pupdate/status.go`: thread resolved blacklist config into detection for status.

## Required changes

1. Extend detection inputs so the resolved folder blacklist config reaches the detector from both `run` and `status`.
2. Skip entering directories whose name matches a configured blacklist entry anywhere in the detection walk, including default shallow scan candidates and configured workspace expansions.
3. Add tests that prove `blah` excludes both `./blah` and nested paths such as `./foo/blah`, while unrelated names remain discoverable.

## Non-goals

- Do not change the existing exact-match blacklist contract into glob or regex matching.
- Do not document the new setting in the README yet.
- Do not broaden detection beyond the already planned `workspace_globs` expansion.

## Acceptance criteria

- Repositories without blacklist config behave exactly as they do today.
- A configured entry such as `blah` prevents detection from entering both `./blah` and `./foo/blah`.
- Non-matching sibling directories remain eligible for detection.
- Detection tests cover default-path exclusions, nested exclusions, and interaction with configured workspace expansions.
